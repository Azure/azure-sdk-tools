
-Goal-
Given a text document that is potentially relevant to this activity and a list of entity types, identify all entities of those types from the text and all relationships among the identified entities.

-Steps-
1. Identify all entities. For each identified entity, extract the following information:
- entity_name: Name of the entity, capitalized
- entity_type: One of the following types: [documentation domain, tooling platform, specification language, code generator, SDK library, programming language, API review tool, release management artifact, workflow automation agent, build pipeline, repository, versioning policy, breaking change governance body, validation rule, package manager, communication channel]
- entity_description: Comprehensive description of the entity's attributes and activities
Format each entity as ("entity"<|><entity_name><|><entity_type><|><entity_description>)

2. From the entities identified in step 1, identify all pairs of (source_entity, target_entity) that are *clearly related* to each other.
For each pair of related entities, extract the following information:
- source_entity: name of the source entity, as identified in step 1
- target_entity: name of the target entity, as identified in step 1
- relationship_description: explanation as to why you think the source entity and the target entity are related to each other
- relationship_strength: an integer score between 1 to 10, indicating strength of the relationship between the source entity and target entity
Format each relationship as ("relationship"<|><source_entity><|><target_entity><|><relationship_description><|><relationship_strength>)

3. Return output in English as a single list of all the entities and relationships identified in steps 1 and 2. Use **##** as the list delimiter.

4. If you have to translate into English, just translate the descriptions, nothing else!

5. When finished, output <|COMPLETE|>.

# Build triggers

## Continuous Integration
Public CI builds run in our Public project, and are triggered automatically whenever a pull request is opened, or a new commit is pushed to an existing pull request, or new commits are merged into the master branch. They run a full build, code linting, unit tests, dependency analysis, and Component Governance detection. They do not run any integration tests.

Rarely, Azure DevOps will miss a pull request trigger, and a CI build will not be queued when it should have been. If the build is Required, it will cause your pull request to be blocked from merging, and you will see a Required build with a status of "Expected — Waiting for status to be reported". In this case, you can manually trigger the build to run on your PR as documented in the next section.

## Manual PR Triggers
Any build pipeline that has enabled pull request validation can be triggered by issuing a bot command in a [comment](https://docs.microsoft.com/en-us/azure/devops/pipelines/repos/github?view=azure-devops&tabs=yaml#comment-triggers) like `/azp run <full name of pipeline including spaces>`. This includes both public CI builds (as above), as well as internal pipelines which have been designed only to be triggered from a bot command issued by a member of the Azure SDK team. These pipelines exist to take actions (such as running live tests) that require access to secrets or to our Azure infrastructure, but that we don't want to risk exposing to untrusted pull requests. 

To create a manually-triggered build, create an internal pipeline with the following options:

![image.png](/.attachments/image-50b79bda-44a5-4b33-a903-d70de7391920.png =500x)

The key settings are:
1. "**_Require a team members' comment before building a pull request_**" must always be checked otherwise this pipeline will automatically trigger which we don't want.
2. "**_Make Secrets available to builds of forks_**" must be checked if this build pipeline requires secrets to run live tests otherwise the pipeline will not have access to the secrets and will fail for any PR submitted from a fork, which is the common scenario. 

### Manually trigger other pipelines

In some cases, changes to one service may need to be tested using pipelines associated with another service. For example, working on files in `sdk/storage` or `sdk/core` and wanting to ensure that other services are not impacted.

To trigger a pipeline against a specific PR, navigate to the pipeline in your browser and click the blue `Run Pipeline` button

<IMG  src="https://user-images.githubusercontent.com/2158838/212986512-c42aa5ae-aaf7-45e8-9697-da927c736f15.png"  alt="image"/>

In the `Branch/tag` field enter `refs/pull/<PR-Number>/merge` (e.g. `refs/pull/4241/merge`): 

<IMG  src="https://user-images.githubusercontent.com/2158838/212986713-d3d8b26a-b788-41a5-80ad-461b131f803b.png"  alt="image"/>

Then click the blue `Run` button to execute the pipeline.

## Scheduled Builds
Internal builds are scheduled to run daily to ensure that factors outside the repo (e.g. build machines, network resources, etc.) don't cause failures. These builds typically run the same set of tasks that CI builds do, but additionally, dailies may also run integration tests or publish documentation updates. 

The specifics of the daily build tasks vary from repo to repo. Daily builds should follow the template convention described in the "Build Pipelines" section with a template YAML and scenario specific YAML files where appropriate for projects that have integration tests. 

The build schedules should be staggered across all 24-hours of the day to spread out workloads as we are a international team and there isn't any time where people aren't working. When the pipeline generation tool creates the pipeline it will try and use the definition id to bucket it across the 24-hour day, and we occasionally will rebalance the schedules as we find hotspots during the day for when a build happens.

For our daily builds we should only schedule them for Mon-Fri to leave the weekends for other weekly or monthly builds. For longer runs or runs we don't need all the time we should schedule them for the weekends.
