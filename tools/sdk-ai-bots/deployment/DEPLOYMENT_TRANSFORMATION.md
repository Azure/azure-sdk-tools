# sdk-ai-bots Deployment Transformation Plan

> Generated by the `chatbot-azd-bicep-deployment-transformer` skill, leveraging
> the existing `../azd-experiments` repo as the source for Bicep modules and
> azd lifecycle hooks.

---

## 1. Executive Summary

**Current state.** `tools/sdk-ai-bots/` is a 6-component chatbot system (Teams
frontend, Go backend, Azure Function, Foundry agent, Knowledge Sync, and
Evaluation framework). Each component has its own Azure DevOps `ci.yml` /
`cd.yml`, but:

- Infrastructure is partially defined in `azure-sdk-qa-bot/infra/azure.bicep`
  (the Teams-Toolkit bot only) and partially provisioned out-of-band (backend
  App Service, Cosmos DB, Search, Key Vault, App Configuration, Logic App,
  Foundry project, AI model deployments).
- Each CD pipeline hard-codes its own variable group (`Azure SDK QA Bot Dev /
Preview / Prod Variables`) and environment list. There is no shared
  environment contract.
- The backend CD rebuilds the Docker image at deploy time (mixing CI + CD).
- The agent CD reuses the existing image for `prod` ("prod reuses existing
  image") тАФ a manual promotion convention, not enforced.
- The Teams bot uses Teams Toolkit's `teamsapp.yml`, which also runs ARM
  deployment as part of `teamsapp provision`, creating a second source of
  truth alongside the backend's portal-configured resources.
- No `azure.yaml`, no `azd` workflow, no `what-if` / drift detection, no
  rollback runbook, no rollout strategy beyond `az webapp restart`.
- The companion `azd-experiments` repo already contains a complete Bicep
  module set (`qaBotSharedResources`, `qaBotAgent`, `qaBotBackend`,
  `qaBotFrontend`, `qaBotFunctionApp`, `qaBotLogicApp`), a working
  `azure.yaml`, and a TypeScript hook pipeline driving the infra-only layers.

**Target state.** Adopt the `azd-experiments` structure in-tree under
`tools/sdk-ai-bots/deployment/`, drive every component through:

- one shared environment-suite (`infra/environments/environment-suite.yaml`),
- one `azure.yaml` orchestrating `azd provision` + per-service `azd deploy`,
- reusable pipeline templates for provision / CI / CD,
- standardized per-component provision / CI / CD pipelines,
- slot- or ring-based rollout + an explicit per-component rollback path,
- runbooks for deploy and rollback.

**Major risks / required decisions.**

1. **Single resource group today.** `azd-experiments/infra/main.bicepparam`
   uses one RG (`azure-sdk-qa-bot`) regardless of environment. Real
   environment isolation requires either per-env RGs or per-env
   subscriptions. Decision required before stage/prod rollout.
2. **Backend image is rebuilt in CD.** Build-once / promote-many requires
   moving `docker build` out of CD into CI. Pipelines below enforce this;
   `prod` CD only re-tags / pulls.
3. **Teams Toolkit ARM deploy overlaps with main Bicep.** The
   `azure-sdk-qa-bot/infra/azure.bicep` covers a subset of resources also
   created by `qaBotFrontend/userAssignedIdentity.bicep`. Path forward:
   retire `teamsapp.yml`'s `arm/deploy` step and let the unified Bicep
   module own those resources; keep `teamsapp.yml` only for manifest
   creation + Teams app publish.
4. **Logic App connections need OAuth consent.** `postprovision.ts` has
   `TODO` stubs for the Teams / Blob managed-API consent flow. This must
   be documented as a one-time manual step per environment.
5. **Key Vault soft-delete name collision.** `azuresdkqabot-keyvalut` (typo
   in original Bicep) is shared. Recommend renaming with environment
   suffix, but flag as breaking change requiring app-side config update.

**Migration strategy.** Three phases:

1. **Land scaffolding (this PR).** Generate the `deployment/` tree, port
   Bicep and hooks from `azd-experiments`, publish reusable pipeline
   templates. Existing CD pipelines remain functional and unchanged.
2. **Cut dev over.** Run `azd provision` end-to-end against a fresh dev
   resource group. Switch dev CD pipelines to consume the new templates
   and the shared environment-suite.
3. **Promote.** Apply to preview, then prod, gated by approvals and
   what-if review.

**Expected benefits.** One source of truth for infra, build-once artifacts,
explicit rollback, lower onboarding cost, and an auditable production
deployment path that cannot be triggered from a developer laptop.

## 3. Target Architecture

### 3.1 Shared environment-suite

A single YAML file (`infra/environments/environment-suite.yaml`) declares
**dev / preview / prod** with per-env subscription, resource group prefix,
region/ring list, approval requirement, and the
`prodDeployOnlyFromPipeline` flag. Every pipeline `template`s a step that
loads this file and exports `AZURE_SUBSCRIPTION_ID`, `AZURE_LOCATION`,
`AZURE_RESOURCE_GROUP`, `APPROVAL_REQUIRED`, etc.

### 3.2 Bicep module model

Ported wholesale from `azd-experiments/infra/`:

- `infra/main.bicep` (subscription scope, creates RG, modules in order
  `shared-resources` тЖТ `agent` тЖТ `frontend` тЖТ `backend` тЖТ `function-app`
  тЖТ `logic-app`).
- `infra/modules/qaBotSharedResources/sharedResources.bicep`
- `infra/modules/qaBotAgent/component.bicep`
- `infra/modules/qaBotFrontend/userAssignedIdentity.bicep` (+ existing
  `azureSdkQaBotModule.bicep`)
- `infra/modules/qaBotBackend/serverfarm.bicep`
- `infra/modules/qaBotFunctionApp/serverfarm.bicep`
- `infra/modules/qaBotLogicApp/logicAppResources.bicep` (+
  `workflowDefinition.json`)
- `infra/environments/{dev,preview,prod}.parameters.json` тАФ generated
  from the environment-suite.

### 3.3 azd usage model

- `azd provision --environment <env> --no-prompt` applies `main.bicep` and
  runs `hooks/postprovision.ts` (the infra-layer pipeline from
  `azd-experiments/hooks/lib/layers.ts`).
- `azd deploy <service> --environment <env> --no-prompt` deploys frontend
  / function-app / agent (the three azd-services in `azure.yaml`). Backend
  is deployed with `az webapp config container set` from CD because it
  uses a slot-swap rollout strategy that azd does not model natively today.

### 3.4 Pipeline model

Per component:

```text
provision pipeline тЖТ CI pipeline тЖТ CD pipeline тЖТ smoke тЖТ rollout тЖТ rollback path
                          тЖС
                  publishes artifact
                          тЖУ
                       CD consumes
```

All pipelines `extends`-template the existing
`/eng/pipelines/templates/stages/1es-redirect.yml` so 1ES Official runs are
preserved.

### 3.5 Rollout model

- **Frontend / backend / agent-server (App Service):** deploy to a
  `staging` slot, smoke-test, swap into `production`. Rollback = swap back.
- **Function App:** Elastic Premium + deployment slot (recommend `staging`)
  with same swap pattern. Currently single slot; provision pipeline adds
  the slot.
- **Logic App:** workflow `state=Disabled` during deploy, smoke-test on
  trigger, then `state=Enabled`. Rollback = restore previous ARM template
  version via `az deployment group create --template-file <previous>`.
- **Foundry hosted agent:** deploy new revision, validate via Responses
  protocol ping, traffic-shift between revisions (when supported) or
  re-deploy previous container tag.
- **Knowledge sync:** scheduled тАФ no rollout, but each run records a
  manifest. Rollback = re-run with previous knowledge snapshot from blob
  versioning.

### 3.6 Rollback model

- Per-component CD must record `LastKnownGoodTag` to App Config
  (`Deployment:<component>:LastKnownGoodTag`) on successful prod deploy.
- `pipelines/templates/rollback.yml` reads that value and re-points the
  App Service / Function App to it (no rebuild). For slot-based components
  it does a slot swap; for revision-based components it shifts traffic.
- `scripts/rollback.ps1` is the local-fallback wrapper.

### 3.7 Security model

- Pipeline-only service connection (`azuresdkqabot-prod`) has Contributor
  on the prod RG.
- Developers have `azd deploy` access to **dev only**, via a separate
  `azuresdkqabot-dev` service connection or local AAD login.
- All app-to-app auth via managed identity (already in current Bicep).
- All secrets in Key Vault with reference from App Service config; no
  secrets in pipeline YAML or repo.
- Branch protection on `main`; PR review required for `tools/sdk-ai-bots/
deployment/**` and `infra/**`.

## 5. Environment-Suite Contract

See [deployment/infra/environments/environment-suite.yaml](infra/environments/environment-suite.yaml).
Highlights:

- Three environments тАФ `dev`, `preview`, `prod`.
- One region per env in phase 1 (`westus2`); `prod` adds `eastus` once
  multi-region resources land.
- `prod` has `approvalRequired: true` and `prodDeployOnlyFromPipeline:
true`.
- Per-env subscription IDs and RG prefix are the **only** values pipelines
  read; everything else flows from there.

```text
1. shared-resources    (managed identity, ACR, Storage, Cosmos, Key Vault,
                        App Config, Search, Log Analytics, action group)
2. agent platform      (App Insights, AI Services, Foundry project, models)
3. frontend identity   (bot user-assigned identity тАФ needed by backend slot)
4. backend             (App Service plan, backend slot, agent slot)
5. function-app        (Elastic Premium, function site + slot)
6. logic-app           (integration account, connections, workflow)
7. hosted agent app    (azd deploy agent тАФ runs on the Foundry project)
8. application images  (frontend / function-app / agent-server containers)
9. traffic activation  (slot swap; Logic App state=Enabled)
10. knowledge sync     (first run after Storage + Search are ready)
```

This is exactly the order encoded in
`azd-experiments/infra/main.bicep` and reproduced in the ported
`infra/main.bicep`.

## 8. Rollout Plan

| Env     | Strategy                                               | Approval | Health gates                                           |
| ------- | ------------------------------------------------------ | -------- | ------------------------------------------------------ |
| dev     | direct deploy to production slot                       | none     | smoke /health 200; warn-only on latency                |

## 9. Rollback Plan

| Component      | Primary                                                         | Fallback                                       | Blocked by                                                     |
| -------------- | --------------------------------------------------------------- | ---------------------------------------------- | -------------------------------------------------------------- |
| frontend       | `az webapp deployment slot swap` (back to previous prod)        | redeploy `LastKnownGoodTag` to production slot | none                                                           |
| backend        | slot swap back                                                  | redeploy `LastKnownGoodTag`                    | breaking Cosmos schema change тЖТ must use schema rollback first |
| function-app   | slot swap back                                                  | redeploy previous tag                          | queue schema change тЖТ drain then roll                          |
| agent-server   | redeploy previous tag (slot is production)                      | n/a                                            | none                                                           |
| hosted agent   | redeploy previous tag via azd                                   | n/a                                            | model deprecation in Foundry project                           |
| logic-app      | re-apply previous ARM template revision                         | manually disable workflow                      | OAuth re-consent if connection was rotated                     |
| knowledge-sync | restore previous Storage blob snapshot + re-run sync against it | None тАФ no traffic exposure                     | Search index schema migration                                  |

**Hard rule:** rollback never rebuilds artifacts. CD records
`LastKnownGoodTag` to App Configuration on every successful prod deploy.

## 13. Assumptions and Open Questions

1. **One subscription per env** (current setup is single subscription with
   per-env RGs). Environment-suite uses placeholder subscription IDs;
   adopters must fill in real values.
2. **Frontend image name** is assumed `azure-sdk-qa-bot` (matches current
   Bicep). If renamed for clarity (e.g. `azure-sdk-qa-bot-frontend`),
   update `azure.yaml` and the frontend CD template together.
3. **Slot count.** Frontend, function-app, and backend get a `staging`
   slot in the new infra. The current backend already has an `authoring`
   slot and the agent server uses the `agent` slot тАФ both are preserved.
4. **Knowledge-sync versioning.** Storage blob versioning is assumed
   enabled. If not, `sharedResources.bicep` must be updated to enable it.
5. **Evaluation framework** is in scope for CI but **out of scope** for
   the deployment transformation тАФ it does not deploy infrastructure.
